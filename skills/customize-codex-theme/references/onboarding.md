# 首次检查与引导

除非用户明确要求其他语言，默认使用简体中文。先运行只读 `doctor`，再按结果选择一种输出；不要把成功与 `NOT_READY` 混写。

## 环境就绪

按以下顺序报告真实字段，不能省略 Node.js，也不能把非空状态说成“干净的初始状态”：

```text
环境检查已通过：
- 操作系统：<Windows/macOS>
- Codex 版本：<version>
- Node.js：<version>（来源：<Codex 内置/CODEX_THEME_NODE/系统 PATH>；Node.js 20+ 与内置 WebSocket 兼容性校验通过）
- 主题状态：<未发现/逐项报告 selected、next、loaded、previous>
- 调试端点：<未发现/已验证>
- 主题启动器：<未发现/逐项报告路径>
- 本次操作：仅完成只读检查，未修改任何文件，也没有重启 Codex
```

只有主题状态、端点和启动器均为空时，才说明“当前处于干净的初始状态”。已有主题或任一非空状态时逐项如实报告，不得宣称干净，并提供继续定制、切换、修复或恢复选项。

随后说明四种入口：

1. **发送参考图片**：分析构图、配色、对比度及宽屏/分栏/窄屏裁切，生成三种预览方向。
2. **描述主题风格**：请用户尽量说明明暗、主色、材质、克制或张扬、背景插画等，再生成三套方向。
3. **让我推荐方案**：信息不足时只问 2–3 个关键偏好，再给三套推荐方向；分别说明配色、字体、界面材质、背景和可读性。
4. **体验六套内置主题**：只有用户选择内置主题或第 4 种入口之后，才读取目录并展示六套简要预览卡；不自动应用，用户选中后再详细预览。

进入准备阶段前必须说明：主题只能通过独立的 **Codex Themes** 启动入口加载；**Codex Original** 启动入口始终打开未注入的原版 Codex，用于回退。询问用户是否要创建这两个主题自有入口，不得静默创建或覆盖现有快捷方式。

结尾请用户回复序号，或直接发送图片/风格描述。

## Node.js 未就绪

若没有可信的 Node.js 20+ 运行时，或缺少内置 WebSocket，输出 `NOT_READY`，报告已发现的操作系统、Codex、Node.js 失败原因及其余可确认状态，并明确本次未修改、未安装、未重启。

- 不得静默或自动安装，也不得先安装再征求同意。
- Windows：`doctor` 报告“没有可信运行时”不等于系统没有 Node.js。先只读解析现有 `node.exe` 的绝对路径，验证版本为 20+ 且内置 `WebSocket` 可用；若适配器支持 `CODEX_THEME_NODE`，仅为当前命令临时指定该已验证路径并重跑 `doctor`，不得持久化环境变量或修改 `PATH`。确实缺失时，只有获得用户当前轮明确授权，才从官方来源安装 Node.js LTS。安装前说明来源、版本、安装范围、`PATH` 与管理员权限影响；安装后重跑版本和 WebSocket 校验。
- macOS：主题适配器只信任 Codex 内置且签名匹配的 Node.js。缺失或签名/兼容性失败时，优先修复、更新或重新安装官方 Codex；不得用 Homebrew 或系统 Node 绕过签名校验。
- 用户拒绝安装或修复时保持 `NOT_READY`，不继续准备或应用主题。
